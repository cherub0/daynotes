# DayNotes 数据安全与项目迁移设计

## 背景

本轮优化先处理数据安全，再继续做使用效率和发布工程。已合并的跨午夜修复解决了“程序运行到第二天后可能把昨天内容覆盖为空”的直接原因，但还缺少用户可见的恢复能力。为了让之后的优化有更稳的基础，本设计先增加单条便签历史版本、整库备份与恢复，并在实施前把项目目录迁移到统一命名的位置。

## 范围

本次包含三部分：

1. 将本地项目从 `D:\for_cherub\daynotes` 移到 `D:\for_cherub\05.codex\codex-daynotes`。
2. 增加便签级历史版本和数据库级备份恢复。
3. 增加对应的设置入口、恢复界面、错误处理和测试。

本次不修改 GitHub 仓库名、Tauri 应用标识、数据库 app data 路径或产品名。GitHub 继续使用 `cherub0/daynotes`，产品继续叫 DayNotes。本次也不处理 SMTP 密码加密，凭据安全作为数据安全主题的后续子任务单独设计。

## 项目迁移设计

迁移前先清理已合并的旧 worktree，尤其是 `.worktrees\share-range-todo-schedule`。清理前必须确认对应分支已合并，worktree 无未提交变更，且路径解析结果仍在当前仓库的 `.worktrees` 目录下。

迁移执行方式：

1. 创建目标父目录 `D:\for_cherub\05.codex`。
2. 将整个仓库目录移动为 `D:\for_cherub\05.codex\codex-daynotes`。
3. 保留 `.git`、现有分支、远端配置、标签、未跟踪用户文件和文档。
4. 在新路径运行依赖和验证命令，确认 Node、Rust、Tauri 配置仍可工作。

迁移不改变数据库位置，所以用户已安装环境中的数据仍在 `%APPDATA%\com.daynotes.app\daynotes.db`。迁移只影响开发工作区路径。

## 数据模型

新增 `note_revisions` 表记录每条便签的历史版本：

```sql
note_revisions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  note_date TEXT NOT NULL,
  content TEXT NOT NULL,
  todos TEXT NOT NULL,
  created_at TEXT NOT NULL
)
```

每次保存便签时，先读取当前数据库里的旧版本。如果旧版本内容和待办与新版本不同，则在更新 `notes` 前写入一条 revision。每个日期最多保留 10 条历史版本，超过后删除最旧记录。相同内容不重复写 revision，避免自动保存制造噪音。

整库备份使用 SQLite backup 方式生成数据库副本，备份目录放在 app data 下的 `backups` 子目录。自动每日备份保留最近 7 天。手动备份可以由用户主动触发，作为单独文件保留，不参与每日自动清理。

## 保存与备份流程

便签保存和 revision 写入放在同一个事务里：

1. 校验日期、内容和待办 JSON。
2. 查询当前 `notes` 记录。
3. 当前记录存在且与新数据不同，则写入 `note_revisions`。
4. 清理该日期超过 10 条的旧 revisions。
5. upsert 当前 `notes` 记录。
6. 提交事务。

如果 revision 写入失败，则这次保存失败并向前端返回错误。这样不会出现“当前便签已覆盖，但恢复点没写进去”的状态。

每日自动备份在应用启动或首次相关数据库操作时检查当天是否已有自动备份。没有则创建一份。自动备份失败不阻塞便签保存，但会记录状态，设置界面显示最近一次备份是否成功。

## 恢复流程

单条便签恢复：

1. 用户打开当前日期的历史版本列表。
2. 列表显示版本时间和内容摘要。
3. 用户选择版本后进入只读预览。
4. 用户确认恢复后，将该版本写回当前日期的 `notes`。
5. 恢复动作本身也遵守保存事务规则，因此恢复前的当前版本会进入历史版本。
6. 前端刷新编辑器、待办列表和保存状态。

整库恢复：

1. 用户从设置中选择备份文件。
2. 后端打开该文件并执行 `PRAGMA integrity_check`。
3. 校验失败则拒绝恢复，当前数据库不变。
4. 校验通过后，先为当前数据库生成一份“恢复前备份”。
5. 使用 SQLite backup API 将选中的备份复制到当前数据库连接。
6. 前端重新加载当前日期、便签内容、日历标记和设置。

整库恢复会影响所有便签和设置，因此 UI 必须要求二次确认，并明确展示备份文件时间。

## 用户界面

设置弹窗新增“数据保护”区域，显示最近自动备份时间、最近备份状态、手动立即备份按钮和整库恢复入口。恢复整库的入口使用文件选择和确认弹窗，不在用户确认前修改数据库。

当前便签增加“历史版本”入口，可以放在日期区域或编辑工具区，具体实现时优先选择与现有布局冲突最少的位置。历史版本弹窗包含版本列表、预览区和恢复按钮。没有历史版本时展示空状态，不把用户引到无效操作。

恢复完成后使用现有 Toast 告知结果。错误信息使用简体中文，避免暴露内部 SQL 细节。

## 错误处理

便签保存失败时保持现有脏状态，不显示“已保存”。revision 清理失败导致事务失败，避免历史版本数量和当前记录出现不一致。

自动备份失败只记录状态，不中断用户编辑。手动备份失败则向用户显示错误。无效备份、损坏备份、结构不兼容备份必须拒绝恢复。

整库恢复失败时保留当前数据库和恢复前备份。如果恢复已经开始但未完成，后端返回错误，前端重新读取当前数据库状态并提示用户不要继续编辑直到再次验证成功。

## 测试策略

Rust 后端测试覆盖：

- revision 去重：相同内容连续保存不产生重复版本。
- revision 保留：同一天最多保留 10 条。
- 保存原子性：revision 写入失败时当前 note 不被覆盖。
- 自动备份：每天最多生成一份自动备份，保留最近 7 天。
- 手动备份：生成独立备份文件，不被自动清理规则删除。
- 恢复校验：坏文件和 integrity check 失败的文件被拒绝。
- 整库恢复：从有效备份恢复后，notes、settings 和 note dates 一致。

前端测试覆盖：

- 历史版本入口、空状态、列表和预览。
- 单条便签恢复确认后刷新编辑器和待办。
- 整库恢复确认、失败提示和成功后重新加载。
- 备份状态在设置中正确展示。

项目级验证：

- 修正 Vitest 不应扫描 `.worktrees` 的问题。
- 运行 `npm test`。
- 运行 `npm run build`。
- 按需运行 Rust 针对性测试。

## 完成标准

- 项目位于 `D:\for_cherub\05.codex\codex-daynotes`，Git 状态、远端和用户未跟踪文件保留。
- 每条便签在覆盖前保留可恢复的历史版本，最多 10 条。
- 数据库每天自动生成一份可校验备份，自动备份保留 7 天。
- 用户可以恢复单条便签，也可以从有效备份恢复整库。
- 备份或恢复失败时，当前可用数据不被静默破坏。
- 自动化验证通过，并且测试不会被旧 `.worktrees` 干扰。
