# 跨午夜便签数据丢失修复设计

## 问题与根因

DayNotes 在 `App` 每次渲染时调用 `getToday()`，并将结果作为 `useNoteSession` 的 `initialDate`。应用跨午夜保持运行后，任意一次重渲染都会让 `initialDate` 从昨天变为今天。

当前会话 Hook 会响应这个变化并加载今天的内容，但不会同步提交 `currentDate`。因此界面形成“日期仍为昨天、内容已经是今天”的不一致状态。今天没有便签时，编辑器又会把程序触发的 `setContent` 当作用户编辑回传，自动保存最终将空文档写入昨天的记录。

隔离复现已验证以下顺序：

1. 加载 `2026-07-14` 的非空便签。
2. 模拟午夜后父组件把 `initialDate` 更新为 `2026-07-15`。
3. 会话内容变为空，但 `currentDate` 仍是 `2026-07-14`。
4. 模拟编辑器同步更新后，后端收到对 `2026-07-14` 的空内容保存。

## 修复设计

采用双层保护，保持现有日期导航交互不变：

1. `useNoteSession` 将 `initialDate` 明确定义为仅初始化参数。Hook 挂载后，即使父组件重新渲染并传入新的日期，也不得自动加载或切换日期。所有后续日期变化必须继续经过 `changeDate`，以复用保存当前内容、串行化持久化和加载目标日期的既有流程。
2. `Editor` 使用 Tiptap `setContent` 同步外部内容时显式设置 `emitUpdate: false`。程序加载或切换日期不得触发 `onChange`；用户键入、格式化和待办编辑仍正常触发自动保存。

不实现午夜自动跳转到今天。自动跳转可能打断正在编辑的用户，也会引入额外的定时器、系统休眠恢复和保存失败处理。用户仍可通过“回到今天”、前后日期按钮或日历主动切换。

## 数据流与错误处理

- 启动：固定启动时日期，加载对应便签。
- 普通重渲染或跨午夜：保留当前日期和当前内容，不发起隐式加载。
- 主动切换日期：先保存脏数据；保存失败时留在原日期并保留可见内容；保存成功后加载目标日期。
- 程序同步编辑器内容：只更新编辑器文档，不进入脏状态或自动保存队列。
- 用户编辑：保持现有两秒防抖自动保存、手动保存和失败重试行为。

数据库结构和已有便签数据不需要迁移。

## 回归验证

新增自动化测试覆盖：

- Hook 在挂载后收到变化的 `initialDate` 时，仍保留原日期与原内容，不请求新日期。
- 编辑器接收新的外部 `content` 时不调用 `onChange`。
- 用户在编辑器内实际修改内容时仍调用 `onChange`。
- 现有日期切换、自动保存、保存失败保护和卸载保存测试继续通过。

完成后运行针对性测试、全部前端测试、ESLint 和生产构建。该修复不修改 Rust 数据库逻辑，但发布前仍执行完整项目验证。

## 完成标准

- 跨午夜后的任意 React 重渲染不会改变当前日期、当前内容或产生空内容保存。
- 程序加载便签不会被记录为用户编辑。
- 主动日期导航和自动保存行为无回归。
- 所有验证命令无失败。
