# DayNotes SMTP 授权码安全存储设计

## 背景

DayNotes 当前的 `EmailSettings.password` 注释写着“encrypted in storage”，但实际保存设置时会把整个 `AppSettings` 序列化为 `settings.app_settings` JSON。这样 SMTP 授权码会以明文形式进入 SQLite 数据库，也会被数据库备份和整库恢复文件一起携带。

数据安全恢复能力已经覆盖便签历史版本、数据库备份和恢复。下一步需要补齐敏感配置的安全边界：笔记和普通设置可以备份，SMTP 授权码不应该进入数据库、备份文件、日志或 UI 错误信息。

## 范围

本次包含：

1. 将 SMTP 授权码从 SQLite 明文设置中迁移到 Windows 系统凭据存储。
2. 保持现有邮件设置、SMTP 测试、手动发送、定时发送和托盘发送流程可用。
3. 兼容旧数据库：如果旧 `app_settings` 中存在明文授权码，自动迁移到安全存储并清空数据库中的明文字段。
4. 在设置界面显示“已安全保存，留空保持不变”的状态，并提供清除已保存授权码的入口。
5. 增加后端和前端测试，覆盖迁移、保存、读取、清除和缺失授权码时的错误提示。

本次不包含：

- 跨设备同步 SMTP 授权码。
- 将授权码打包进数据库备份。
- 修改 GitHub 仓库名、Tauri 应用标识、产品名或 AppData 路径。
- 引入主密码、账户系统或云端密钥管理。

## 方案选择

### 推荐方案：Windows Credential Manager

使用系统凭据存储保存 SMTP 授权码，数据库只保存邮箱服务器、端口、用户名、收件人、启用状态和主题等非秘密配置。运行邮件功能时，后端按固定服务名和账号键从系统凭据存储读取授权码。

优点：

- 授权码不进入 SQLite，因此也不会进入数据库备份文件。
- 凭据由 Windows 用户会话和系统安全机制管理，符合当前 Windows 桌面应用定位。
- 用户恢复数据库到新机器后，笔记和普通设置可恢复，授权码需要重新输入，安全语义清晰。

代价：

- 授权码不随备份迁移。换电脑、重装系统或恢复到另一个 Windows 用户后，需要重新填写 SMTP 授权码。
- 后端需要抽象凭据存储接口，测试环境使用内存实现，生产环境使用系统实现。

### 备选方案：DPAPI 加密后仍存数据库

使用 Windows DPAPI 加密授权码，再把密文保存在 SQLite。它可以避免纯明文，但密文仍在数据库和备份文件中，并且同样通常绑定机器或用户，跨设备恢复仍可能不可用。这个方案还会增加 Windows API 细节和错误处理复杂度。

### 不采用方案：应用本地密钥加密

如果密钥和数据库都放在应用数据目录，攻击者拿到备份或用户目录时往往也能拿到密钥。它会制造“看起来加密了”的错觉，但实际安全边界较弱，因此不采用。

## 数据模型与序列化

`EmailSettings` 新增只读状态字段：

```ts
password_saved?: boolean
```

Rust 侧对应字段使用默认值兼容旧 JSON。数据库中的 `email.password` 字段仍保留为空字符串，以维持现有结构兼容，但后端保存设置前必须清空该字段。

返回给前端的设置规则：

- `email.password` 始终返回空字符串。
- `email.password_saved` 由系统凭据是否存在实时推导。
- 如果旧数据库中存在明文 `email.password`，读取设置时执行迁移：写入系统凭据存储，随后把数据库中的 `email.password` 清空。

保存设置规则：

- 如果前端传入非空 `email.password`，后端把它写入系统凭据存储，并清空数据库中的 `email.password`。
- 如果前端传入空 `email.password`，后端不覆盖已有系统凭据。
- 保存普通邮件字段时不要求重新输入授权码。
- 清除授权码通过独立命令完成，避免“空字符串到底是保留还是删除”的歧义。

系统凭据键：

- service: `com.daynotes.app.smtp`
- account: `smtp_password`

当前版本只维护一组 SMTP 配置，因此不按邮箱用户名拆分凭据键。这样用户修改发件邮箱后，如果授权码也变化，需要在设置中输入新的授权码；如果未输入，新邮件测试会使用现有保存的授权码并按 SMTP 返回结果提示。

## 后端设计

新增一个小型安全设置模块，负责凭据存储边界：

- `EmailCredentialStore` trait：定义 `get_password`、`set_password`、`delete_password`。
- 生产实现：调用 Windows Credential Manager 或对应 Rust keyring 封装。
- 测试实现：内存存储，不访问真实系统凭据。

新增或调整后端流程：

1. `parse_app_settings` 继续只负责 JSON 解析和默认值填充，不访问外部凭据。
2. `get_settings` 读取数据库设置后，调用迁移/净化流程：
   - 如果 JSON 中存在非空 `email.password`，写入凭据存储。
   - 成功后将数据库中的 `email.password` 清空。
   - 返回给前端时填充 `password_saved`。
3. `save_settings` 接收前端设置：
   - 非空 password 写入凭据存储。
   - 写库前清空 password，并不持久化 `password_saved` 状态。
4. `clear_email_password` 删除系统凭据，返回最新设置状态。
5. 邮件发送和 SMTP 测试统一使用 `resolve_email_settings_for_send`：
   - 从数据库读取普通设置。
   - 从凭据存储读取授权码。
   - 若启用邮件但凭据不存在，返回中文错误“未保存 SMTP 授权码，请在设置中重新填写”。

错误处理要求：

- 不在错误信息、日志或测试输出中打印授权码。
- 迁移失败时不删除旧数据库明文，避免用户失去授权码；但返回设置时仍不把明文传给前端。
- 发送邮件时继续使用现有 SMTP 错误脱敏逻辑。

## 前端设计

设置弹窗中的授权码区域调整为：

- 没有已保存授权码：显示“授权码（非邮箱密码，QQ 邮箱需在设置中生成）”。
- 已保存授权码：显示“授权码已安全保存，留空保持不变”。
- 输入框仍为 `type="password"`，加载设置后值为空。
- 已保存授权码时显示“清除已保存授权码”按钮。
- 点击清除后调用后端命令，成功后更新本地设置状态并显示 Toast。

前端校验调整：

- 如果 `password_saved` 为 true，保存或测试时允许 password 为空。
- 如果 `password_saved` 为 false，且当前输入 password 为空，提示用户输入 SMTP 授权码。
- 用户输入新授权码后，保存和测试都使用这次输入。

## 备份与恢复语义

数据库备份不包含 SMTP 授权码。恢复数据库后：

- 如果恢复到同一个 Windows 用户环境，系统凭据仍在，`password_saved` 继续为 true。
- 如果恢复到另一台机器、另一个用户或系统凭据已被清除，`password_saved` 为 false，用户需要重新输入授权码。
- 旧备份中如果包含明文 `email.password`，第一次读取设置时执行迁移并清空数据库明文。

这让备份文件可以安全分享给用户自己保存，不会意外携带邮箱授权码。

## 测试策略

Rust 后端测试：

- 解析旧 JSON 时保留兼容，但净化保存时清空 `email.password`。
- 旧明文授权码迁移到内存凭据存储后，数据库中的 password 被清空。
- 保存非空 password 会写入凭据存储，数据库不保存明文。
- 保存空 password 会保留已有凭据。
- 清除命令删除凭据，并让返回状态显示 `password_saved = false`。
- 发送邮件解析阶段在缺少授权码时返回中文错误，不尝试 SMTP 连接。
- 迁移写入凭据失败时不删除数据库旧明文。

前端测试：

- 已保存授权码时显示安全保存提示，保存按钮不要求重新输入授权码。
- 未保存授权码时，空 password 触发校验提示。
- 输入新授权码后可以保存。
- 清除授权码按钮调用后端并更新提示。

项目级验证：

- `npm test`
- `npm run build`
- `cargo test`

## 完成标准

- SQLite `settings.app_settings` 不再持久化 SMTP 授权码明文。
- 旧数据库中的明文授权码可以自动迁移到系统凭据存储。
- 邮件测试、手动发送、定时发送和托盘发送仍能读取授权码并工作。
- 用户能看出授权码是否已安全保存，也能主动清除它。
- 数据库备份和恢复的行为清晰：备份不携带授权码，必要时重新输入。
- 自动化验证通过。
